Slide 3

Spec-based development in one sentence

Define the system precisely enough that code generation, human review, validation, and future reimplementation all operate against the same source of truth.

What the spec is

  • A behavioral contract
  • An architecture guide
  • A decomposition of work
  • A set of validation invariants

What the spec is not

  • A vague vision statement
  • A backlog placeholder
  • A loose prompt for a model to improvise from
  • A post-hoc description of already written code

Slide 4

The four-phase lifecycle

1

Specify

Define user-facing behavior, business rules, invariants, acceptance criteria, and runtime constraints.

2

Plan

Translate intent into architecture, data flow, state ownership, orchestration, and failure handling.

3

Task

Break the plan into atomic, parallelizable, testable work units that an engineer or model can execute.

4

Implement

Generate or write the code, then validate it against the spec with builds, tests, and runtime checks.

This is the structure now used across the booking specs: product, API, design, multi-step, and conversational.

Slide 5

Case study: the booking journey forced the spec to mature

Early assumption

“Just build the flow”

Initial implementation pressure makes teams accept under-specified behavior.

Reality

Backend statefulness surfaced everywhere

Selections, defaults, repricing, polling, restore, and session continuity all mattered.

Learning

The spec had to become operational

Not “what the UI looks like”, but exactly how payload, selection, and validation work.

Key shift

The useful artifact stopped being “a design plus some tickets” and became a family of coordinated specs: product flow, API contract, design system, multi-step implementation contract, and conversational implementation contract.

Slide 6

What we learned from the booking implementation

1. The payload is the product

The evolving booking payload is the real state machine. UI is just a projection of it.

2. “Selected” matters

Backend-selected choices must become the default everywhere. Array order is not a contract.

3. Receipt is the pricing truth

Selections are not valid until repriced successfully. UI optimism must be constrained.

4. New stay = new decision tree

Changing the calendar selection clears downstream products and restarts the journey.

5. Async flows need contracts

Flights and cars required explicit task-group polling, statuses, retries, and validation.

6. Restore is part of the feature

Session headers, URL snapshots, and rehydration are product behavior, not implementation detail.

Slide 8

Specs now need to be model-readable

Deterministic language

Say “must”, “must not”, and “fallback order”, not “ideally” or “roughly”.

Operational invariants

State what happens on failure, reload, retries, and conflicting choices.

Canonical ownership

Define what the backend owns, what the frontend owns, and what the model is allowed to do.

A good AI-era spec does not just describe the happy path. It constrains interpretation.

Slide 9

The new role of engineering

Less time on

  • boilerplate generation
  • repeating known UI patterns
  • mechanical wiring once contracts are clear

More time on

  • state design
  • failure modeling
  • API contract rigor
  • validation boundaries
  • quality of the executable spec

Practical reframing

The engineer becomes the author of constraints, invariants, and verification loops, not just the author of source code.

Slide 11

Recommended working model for teams

Step 1

Write the spec family first

Product, API, design, runtime mode, and validation constraints.

Step 2

Implement in vertical slices

But wire the real backend early so the spec gets corrected by reality.

Step 3

Feed learnings back into the spec

Every runtime discovery becomes a sharpened contract.

Step 4

Treat the spec as reusable capital

The next model, team, or interface mode should be able to rebuild from it.

Slide 12

Bottom line
