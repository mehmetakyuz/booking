An agile take on a waterfall-y idea — write specs not up front, but as you discover what matters.

The problem

Vibecoding doesn't reproduce well

✗ Prompt-only approach
  • Each session starts cold
  • Edge cases discovered, then forgotten
  • Model improvises where you haven't specified
  • Output drifts across models & runs
  • Quality is prompt-author-dependent
✓ Spec-as-you-go approach
  • Discoveries become permanent constraints
  • Edge cases encoded once, applied everywhere
  • Model has a contract to implement against
  • Consistent output across models & runs
  • Any engineer can pick it up and continue
The spec is not planning ahead — it is recording what you've already learned so the next run doesn't start from zero.

spec-design.md

Design language distilled into constraints

Tokens — not vibes
  • --brand-primary: #ff791a — orange, CTA & hero
  • --brand-secondary: #0098a8 — turquoise, helper
  • --brand-black: #17171a — nav rail & dark surfaces
  • Source Serif Pro → display moments only
  • Source Sans Pro → all UI text
Component anatomy
  • Dark top rail, light page canvas
  • White content surfaces, not grey
  • Two-column desktop: step left, receipt right
  • One shared loading pattern — not bespoke per step
Why this matters for AI generation

Without a design spec, models default to generic SaaS UI. With it, every implementation looks and feels like the same product.

/* Bad — model improvises */ .btn { background: blue; } /* Good — spec-encoded token */ .btn-primary { background: var(--brand-primary); /* #ff791a — never deviate */ }

Brand intent encoded as rules: "must not feel like generic SaaS checkout UI."

spec-api.md

Real edge cases we discovered & encoded

Money units

All prices are integers in minor units (pence). 15630000 = £156,300.

// Divide by 100 at // render boundary ONLY. // Never mutate stored value.
Backend-selected defaults

If a response marks a choice as selected: true, that is the default. Array order is not a contract.

New stay = reset

Changing the calendar selection clears all downstream product selections and restarts the decision tree.

Price is a scalar

price { ... } is a query error — HTTP 400. Select price fields as leaf nodes, not objects.

Session continuity

Unique x-tb-sessionid generated per booking, persisted in URL state so restore continues the same backend session.

Receipt = truth

Selections are not valid until a receipt reprice succeeds. Failed receipt errors block progression to the next step.

Each of these was discovered by hitting the live API — not invented upfront. Writing them down made every subsequent implementation correct from the start.

Applying the specs

The implementation contract

Before coding
  • Read all four specs in order
  • Create an explicit task plan
  • Include loading, error, restore, empty states
  • Wire the real API early — let it correct the spec
During coding
  • Treat the spec as the source of truth, not the prompt
  • Update the spec if the live API reveals a gap
  • Track skipped work explicitly — not as a mental note
After coding
  • A passing build is not done
  • Run the app, validate every panel interactively
  • Check CSS class names match between components and stylesheets
  • Verify async flows (flights, cars) end-to-end
Spec maintenance rule

If implementation behavior changes, update the spec in the same commit. The spec is part of the product contract — not optional documentation.

Spec quality

Deterministic language is the whole game

✗ What we started with
  • "Use the real API"
  • "Build a multi-step booking UI"
  • "Make it look like the brand"
  • "Show receipt updates"
  • "Handle async searches"
  • "Let users restore their session"
✓ What the spec says now
  • "Price fields are scalar integers. Divide by 100 at render — never earlier."
  • "selected: true is the default. Array order is not a contract."
  • "A new stay must clear downstream products."
  • "Receipt errors must block step progression."
  • "Session ID must survive URL restore."
must / must not fallback order, not just defaults failure behavior, not just happy path what the backend owns vs frontend

Takeaways

What we'd do again

Start with vibing
  • Don't spec upfront — you don't know what matters yet
  • Build the happy path first to learn what the API actually does
  • Use the friction as signal
Encode the three layers
  • Design language — tokens, not descriptions
  • API contract — invariants, edge cases, payload lifecycle
  • Product behavior — flows, criteria, conditional logic
Make it model-readable
  • Use must / must not — not "ideally" or "roughly"
  • Specify failure behavior, not just success
  • Include a validation phase — a build is not done
  • Update the spec when reality contradicts it
