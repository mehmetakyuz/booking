An Experiment

Spec-Based
Development

Build iteratively. Persist the context as a spec. Let AI models re-implement it independently.

The Hypothesis

What if the context built up during development was precise enough for any AI model to re-implement the same product independently?

The spec is the persisted context.
The code is a consequence of it.

How it worked

The spec is context, persisted

Step 1

Develop iteratively

A working implementation was built through iteration — decisions made, APIs discovered, edge cases resolved. Wired to the live API throughout.

→
Step 2

Persist the context

Each decision, constraint, and API behavior discovered during development was written down as a spec. The spec captures what was learned, not what was imagined.

→
Step 3

Let models re-implement

The spec — not the code — is handed to each AI model. Every implementation is independent. The only variable is the model.

Why this order matters

A spec written without building is just a guess

Pure spec-first

Waterfall in disguise

Writing a spec before touching the system assumes you already know how it will behave. You don't. Real systems surface surprises the moment you engage with them. The spec becomes aspirational fiction — correct in theory, wrong in practice.

VS
Implementation-driven

Agile by design

Each iteration teaches you something you couldn't have known upfront. The spec is updated as you go. It stays accurate because it reflects what was built and tested, not what was imagined.

Full API documentation with every query, mutation, type definition, payload contract, and worked examples — including non-obvious invariants discovered during development.

Complete design language system — color palette, typography scale, spacing grid, every component pattern. No Figma file needed.

The spec defines the pattern. Every model implements it. No discussion needed.

Observations

What spec-based development reveals

Precision

Ambiguity is the enemy

Vague specs produce divergent implementations. Every edge case documented is one fewer thing a model has to invent.

Repeatability

Same input, comparable output

The same spec handed to different models produces implementations that can be directly compared and evaluated.

Validation

Compile ≠ correct

Phase 5 (browser validation against the live API) is what separates a working implementation from one that just builds.
